# Contributing to LegalEase AI

Thank you for your interest in contributing to LegalEase AI! We welcome contributions from the community.

## Getting Started

1. Fork the repository
2. Clone your fork locally
3. Install dependencies: `npm install`
4. Create a `.env.local` file with your API keys (see `.env.example`)
5. Start the development server: `npm run dev`

## Development Process

### Before You Start

- Check existing issues and pull requests
- Create an issue to discuss major changes
- Follow our code style and conventions

### Making Changes

1. Create a feature branch: `git checkout -b feature/your-feature-name`
2. Make your changes
3. Test your changes thoroughly
4. Commit with clear, descriptive messages
5. Push to your fork
6. Create a pull request

### Code Style

- Use TypeScript for type safety
- Follow ESLint rules: `npm run lint`
- Use Tailwind CSS for styling
- Follow existing component patterns
- Write clear, self-documenting code

### Commit Messages

Use clear, descriptive commit messages:
- `feat: add document generation feature`
- `fix: resolve chat interface bug`
- `docs: update API documentation`
- `style: improve button component styling`

## Types of Contributions

### Bug Reports

- Use the issue template
- Include steps to reproduce
- Provide system information
- Include screenshots if relevant

### Feature Requests

- Describe the problem you're solving
- Explain your proposed solution
- Consider alternative approaches
- Discuss potential impact

### Code Contributions

- Focus on one feature/fix per PR
- Include tests when applicable
- Update documentation
- Follow existing patterns

### Documentation

- Fix typos and improve clarity
- Add examples and use cases
- Update API documentation
- Improve setup instructions

## AI and Legal Considerations

### AI Prompts

- Test prompts thoroughly
- Ensure compliance with legal guidelines
- Focus on Indian business law
- Include appropriate disclaimers

### Legal Content

- Verify accuracy of legal information
- Include proper disclaimers
- Focus on educational content
- Avoid providing specific legal advice

## Pull Request Process

1. Update documentation if needed
2. Add tests for new functionality
3. Ensure all tests pass
4. Update CHANGELOG.md
5. Request review from maintainers

### Review Criteria

- Code quality and style
- Test coverage
- Documentation updates
- Performance impact
- Security considerations

## Community Guidelines

- Be respectful and inclusive
- Follow our Code of Conduct
- Help others learn and grow
- Share knowledge and best practices

- Create an issue for questions
- Join our community discussions
